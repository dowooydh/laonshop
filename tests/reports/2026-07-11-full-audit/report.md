## 5. 발견 결함

### P1-01 재고 합계를 초과한 주문이 결제창까지 진행됩니다 - E2E/DB 확정

재현:

1. QA 전용 상품을 `stock=1`, 사이즈 `S,M`으로 생성합니다.
2. S 1개와 M 1개를 각각 장바구니에 담습니다.
3. 체크아웃에서 구매 동의 후 카드결제를 누릅니다.

기대 결과: 같은 상품의 총 주문수량 2가 재고 1을 초과하므로 서버가 주문 생성을 거부해야 합니다.

실제 결과: KSPAY 결제창 로딩 단계까지 진행됐고, DB에 `PENDING`, 총수량 2, 총액 2,000원의 주문이 생성됐습니다. 상품 재고는 1 그대로였습니다.

원인 후보:

- `app/checkout/actions.ts:55-69`가 장바구니 라인별 `stock < qty`만 비교하며 상품별 합계를 만들지 않습니다.
- `app/checkout/actions.ts:71-76`은 요청의 `size`를 상품의 허용 사이즈와 대조하지 않습니다.
- 저장소 전체에서 결제 완료 시 재고 차감·예약·복구 로직이 없습니다.

권고 회귀:

- 동일 상품/동일 사이즈 합산, 동일 상품/다른 사이즈 합산, 두 사용자 동시 주문을 모두 검증합니다.
- 서버 트랜잭션에서 재고 확인과 예약/차감을 원자적으로 처리합니다.
- 요청 사이즈가 상품 `sizes` 목록에 없으면 거부합니다.

### P1-02 KSPAY 승인키와 승인 결과가 평문 HTTP로 전송되며 timeout이 없습니다 - 코드 확정

- `lib/kspay/kspay-provider.ts:29`의 승인 URL이 `http://kspay.ksnet.to/.../recv_post.jsp`입니다.
- `lib/kspay/kspay-provider.ts:145-155`에서 `reCommConId` 승인키를 해당 URL로 전송합니다.
- 같은 fetch에 timeout, 상태 코드 검사, 제한된 재시도, 사용자 복구 경로가 없습니다.
- `lib/kspay/webfep.ts:51-74`의 실승인 fetch에도 timeout이 없습니다.

영향: 네트워크 중간자 노출·변조 위험과 서버리스 함수 장기 점유가 있으며, 사용자가 재시도해 중복 승인 경합을 유발할 수 있습니다. KSNET의 HTTPS 지원 URL과 필수 프로토콜을 사업부에 확인한 뒤 TLS와 bounded timeout을 적용해야 합니다.

### P1-03 중복 result 요청에서 실제 승인 주문이 FAILED로 남을 수 있습니다 - 코드 경합 확정

`app/api/pg/kspay/result/route.ts:15-55`는 다음 순서입니다.

1. 주문이 `PENDING`인지 읽습니다.
2. DB 락이나 처리중 상태 없이 외부 PG 승인을 호출합니다.
3. 성공과 실패 모두 `where status=PENDING`으로 최종 상태를 선점합니다.

동일 결과가 동시에 두 번 들어와 하나는 PG 성공, 다른 하나는 중복/오류 응답이면 실패 요청이 먼저 `FAILED`를 기록할 수 있습니다. 이후 성공 요청의 update는 0건이 되어 실제 승인과 DB 상태가 불일치합니다. 주석의 멱등 보장은 동시 요청까지 성립하지 않습니다.

권고 회귀:

- 승인 처리 전 `PROCESSING` 선점 또는 DB advisory/row lock을 사용합니다.
- 동일 `reCommConId`/주문에 외부 승인 호출이 한 번만 일어나는지 병렬 테스트합니다.
- 실패가 성공 상태를 이기지 못하고, 불명확 결과는 조회/운영 보정 대상으로 남는지 확인합니다.

### P1-04 새 주문 생성에 서버 멱등키가 없습니다 - 코드 확정

`app/checkout/actions.ts:175-187`은 요청마다 새 moid와 주문을 즉시 생성합니다. UI의 `pending`은 한 탭의 빠른 더블클릭은 막았지만, 두 탭·재전송·네트워크 재시도에는 공통 멱등키가 없습니다. 동일 장바구니가 복수 주문/복수 결제창으로 진행될 수 있습니다.

단일 탭 더블클릭은 주문 1건으로 통과했습니다. 다중 탭 병렬 제출은 이번 실행에서 별도 자동화가 없어 미실행했으며, 서버 코드상 보호 장치가 없음을 결함 근거로 삼았습니다.

### P2-01 동일 이메일 동시 가입 한 요청이 Prisma P2002와 500 화면으로 끝납니다 - E2E 확정

두 탭에서 같은 신규 이메일로 동시에 가입했습니다. 사용자 1명만 생성됐지만 한 탭은 홈으로 이동하고 다른 탭은 일반 오류 화면을 표시했습니다. 서버 로그는 `P2002 Unique constraint failed on fields: (email)`이었습니다.

`app/(auth)/actions.ts:33-39`가 `findUnique` 후 `create`하는 check-then-act 구조이며 unique 충돌을 사용자 오류로 변환하지 않습니다. `create`의 P2002를 잡아 `이미 가입된 이메일입니다.`로 반환해야 합니다.

### P2-02 이메일 대소문자만 다른 계정이 중복 생성될 수 있습니다 - DB 롤백 검증

테스트 트랜잭션에서 동일 이메일의 대소문자 변형 2개가 모두 insert됐고 전체 트랜잭션은 강제 rollback해 잔존 0건을 확인했습니다.

- `app/(auth)/actions.ts:12`, `33`, `66`은 이메일을 소문자 정규화하지 않습니다.
- `prisma/schema.prisma:18`은 일반 `String @unique`라 PostgreSQL에서 대소문자를 구분합니다.
- 로그인 실패 잠금 key만 소문자로 변환해 두 계정의 잠금이 서로 영향을 줄 수 있습니다.

저장 전 trim/lowercase 정규화와 DB의 case-insensitive unique 제약을 함께 적용해야 합니다.

### P2-03 320px 로그인 헤더가 수평 스크롤됩니다 - 실화면 확정

### 주문·결제·사후처리

- 회원 배송지 프리필과 구매 동의 전 버튼 비활성, 배송지 누락 오류를 확인했습니다.
- 카드결제의 단일 탭 빠른 더블클릭에서 주문 1건만 생성됐습니다.
- 주문 가격은 상품 DB 가격으로 재계산됐고 주문 총액과 항목 합계가 일치했습니다.
- KSPAY 폼에서 jQuery가 KSPAY 스크립트보다 앞서며 로딩 화면까지 진입했습니다.
- callback hostile 문자열은 `<`가 `\u003c`로 이스케이프돼 script 주입이 없었습니다.
- result 빈 POST는 주문을 실패 처리하지 않고 기존 PENDING을 유지했습니다.
- PENDING 재결제는 새 주문을 만들지 않고 동일 주문을 재사용했습니다.
- 타 사용자 주문 URL은 404였고 승인번호·배송정보가 노출되지 않았습니다.
- 합성 PAID 상태에서 주문완료, 영수증 링크, 결제 직후 장바구니 비움, 취소·반품 1회 접수를 확인했습니다.
- 비허용 계정의 원클릭/수기 실승인 경로는 준비중 오류로 차단됐습니다.
- 등록카드 DB에는 mock token과 마스킹 번호만 남고 원문 카드번호가 없음을 확인했습니다.

## 7. 모바일·Android·접근성

- 1440x900과 390x844에서 홈, 목록, 상세, 장바구니, 체크아웃의 가로 overflow가 없었습니다.
- 320x568 로그인 헤더에서만 overflow를 재현했습니다.
- Android Chrome 133에서 홈과 검색 페이지가 실제 렌더됐습니다.
- Android 후속 자동조작 중 Chrome 자체 ANR이 발생했습니다. exit-info에 Chrome renderer 약 252~259MB와 ANR trace가 있었고 앱 프로세스 종료는 QA 복구를 위한 force-stop 기록이 대부분입니다. LAONSHOP의 제품 크래시로 단정하지 않았습니다.
- 전역 `:focus-visible`, 본문 바로가기, reduced-motion 분기는 코드에서 확인했습니다.
- 실제 키보드 전체 순회와 200% 텍스트 확대는 도구 한계로 완료하지 못했습니다.

## 8. HTTP·오류 경계

- `/api/pg/kspay/callback`, `/api/pg/kspay/result`의 GET은 405입니다.
- 보호 페이지는 Next 스트리밍 응답이 HTTP 200이어도 본문에 `NEXT_REDIRECT ... /login;307`을 포함하고 브라우저는 로그인으로 이동합니다.
- 존재하지 않는 상품도 초기 HTTP는 200일 수 있으나 본문 `NEXT_HTTP_ERROR_FALLBACK;404`와 브라우저 404 UI를 확인했습니다.
- 로그인 잠금은 UI 메시지 기반이며 HTTP 429를 반환하지 않습니다.
- Neon cold wake-up 때 첫 동적 요청이 약 15.55초 걸리고 일시적 `PostgreSQL connection: Closed` 로그 2건 후 200으로 복구됐습니다. 최종 direct DB 기준선 조회도 한 차례 약 65초 지연 후 정상 완료됐습니다. 문서화된 무료 티어 환경 리스크로 분리했습니다.

## 9. 미실행·제한

- 실카드 승인, 실제 자동취소, 실영수증 조회는 실행하지 않았습니다.
- 운영 Vercel 배포와 운영 DB는 변경·검증하지 않았습니다.
- Safari/WebKit, iOS 실제 기기는 검증하지 않았습니다.
- 실제 PG timeout/503, 중복 승인 콜백은 안전상 fault injection하지 않았습니다.
- Android에서 홈·검색 이후 전 기능은 Chrome ANR로 PARTIAL입니다.
- 200% 글자 확대와 전 구간 키보드 순회는 완료하지 못했습니다.
- 관리자 화면은 저장소에 존재하지 않아 검증 대상이 없습니다.
